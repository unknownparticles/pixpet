# 设计：Focus Pet MVP 产品基础

## 背景

PixPet 的公开信息显示，它把番茄钟、专注奖励、像素宠物收集、商店、抽奖、地图、任务、排行榜和平台能力组合在一起。相邻产品也说明：当“可爱陪伴体 + 自我管理”足够简单、可分享时，产品有机会形成传播。

当前仓库还没有实现代码。因此本变更只定义产品意图、MVP 行为和后续工程边界，不直接创建 Flutter 应用。

## 已确认决策

- 技术栈：Flutter。
- 首发策略：iOS-first。
- 专注模式：MVP 先做 soft-focus，不做硬拦截。
- 视觉资产：MVP 使用生成式像素资产，必须保证原创，后续可替换为手绘资产。
- 数据策略：MVP 本地优先，不要求账号和云同步。

## 产品闭环

MVP 围绕一个核心闭环：

1. 用户开始专注。
2. App 进入 soft-focus 全屏计时状态。
3. 用户完成专注并获得能量。
4. 能量喂养或升级宠物。
5. 用户看到下一个确定性解锁目标。
6. 用户保存或分享学习成果卡片。
7. 用户因为提醒、宠物状态或未完成解锁再次回来。

这个闭环故意小于 PixPet 的完整公开功能面。目标是先验证情绪激励和留存，再追加地图、战斗、云同步或社交系统。

## 能力边界

- 产品策略：定义目标用户、差异化、爆款假设和成功指标。
- Focus Pet MVP：定义用户可见行为。
- 后续实现变更：再拆客户端架构、本地存储、资产生成流程、数据埋点和平台权限。

## 关键设计选择

- 使用确定性奖励，不在 MVP 做付费随机抽卡。
- 本地优先，不要求账号。
- 使用 soft-focus 作为首版专注约束：全屏计时、温和提醒、放弃确认、完成反馈。
- 宠物反馈鼓励用户，不羞辱用户。
- 分享卡默认不展示用户输入的私密学习任务。
- 生成式像素资产先满足 MVP 验证，但文件命名、尺寸和主题要为后续正式美术替换留出空间。

## 风险

- 宠物成长太慢会削弱首次完成后的奖励感。
- 过早接入硬拦截会被平台权限和审核复杂度拖慢。
- 游戏系统过重会让产品变成新的分心源。
- 生成式资产如果风格不统一，会削弱产品质感；实现阶段需要定义统一提示词和资产规范。

## 后续实现指导

后续实现应创建一个小而清晰的 Flutter MVP。模型和服务要让初学者能读懂：

- `FocusSession`：记录一次专注会话。
- `PetProfile`：记录宠物状态。
- `RewardLedger`：处理能量发放和确定性解锁。
- `ShareCard`：生成或承载分享卡数据。
- `DistractionControl`：只抽象 soft-focus 能力，不接硬拦截。

本 proposal 阶段不实现这些文件。
